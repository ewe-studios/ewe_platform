# JsonHash Derive Macro

## Iron Law: Zero Warnings

> **All code must compile with zero warnings and pass all lints. No suppression. No exceptions.**
>
> - `cargo clippy -p foundation_macros -- -D warnings -W clippy::pedantic` — zero warnings
> - `cargo doc -p foundation_macros --no-deps` — zero rustdoc warnings
> - `cargo test -p foundation_macros` — zero compilation warnings
> - **No `#[allow(...)]`, `#[expect(...)]`, or `#![allow(...)]` anywhere.** Fix the code, never suppress.

## Overview

Create a derive macro `JsonHash` that generates a `struct_hash()` method for structs. The hash is computed from:
1. The name of the struct
2. The serialized JSON representation of the struct

This macro will be applied to:
- All generated resource types for all providers
- All client function argument structs (excluding `*_builder` and `*_execute` functions)

## Motivation

### Why JsonHash?

1. **Resource Identification**: Unique, reproducible hashes for API resources enable:
   - Change detection between deployments
   - Deduplication of identical resources
   - Efficient caching and comparison

2. **State Store Integration**: The deployment system needs to track:
   - What version of a resource is deployed
   - Whether a resource has changed since last deployment
   - Quick equality checks without deep comparison

3. **Deterministic Hashing**: By including both struct name and JSON content:
   - Same struct type with same values = same hash
   - Different struct types with same values = different hashes
   - JSON serialization ensures consistent field ordering

### Why This Design?

**Struct name + JSON (not just struct name):**
- Allows detecting changes in resource configuration
- Two `Service` structs with different `name` fields produce different hashes

**JSON serialization (not Debug output):**
- JSON is stable and well-defined
- Debug output varies with formatting and implementation details
- JSON already required via `Serialize` trait

**`struct_hash()` method (not a trait impl):**
- Simple, direct API
- No additional trait bounds for callers
- Generated code is self-contained

## Requirements

### Core Requirement: Serialize Dependency

Any struct using `#[derive(JsonHash)]` **must** also derive `Serialize` from serde. The macro should:

1. **Not automatically add `Serialize`** — users must explicitly derive it
2. **Emit a clear compile error** if `Serialize` is missing
3. **Use `serde_json`** for serialization (already a dependency)

```rust
// VALID: User explicitly derives Serialize
#[derive(JsonHash, Serialize)]
pub struct MyResource {
    pub name: String,
    pub value: i32,
}

// INVALID: Missing Serialize - should produce compile error
#[derive(JsonHash)]  // ERROR: JsonHash requires Serialize
pub struct MyResource2 {
    pub name: String,
}
```

### Hash Computation Algorithm

```
┌─────────────────────────────────────────────────────────┐
│  1. Get struct name as string: "MyResource"             │
│  2. Serialize struct to JSON: {"name":"test",...}       │
│  3. Concatenate: "MyResource{...}"                      │
│  4. Compute SHA-256 hash                                │
│  5. Encode as base85 string                             │
│  6. Return hash string                                  │
└─────────────────────────────────────────────────────────┘
```

**Why SHA-256 + base85?**
- `foundation_macros` already uses `sha2` and `base85rs` for `EmbedFileAs`/`EmbedDirectoryAs`
- Consistent with existing hashing in the codebase
- Base85 is more compact than hex (20% smaller)

### Application Scope

The macro should be applied to:

#### 1. All Generated Resource Types

Every struct generated by `gen_resource_types` gets `JsonHash`:

```rust
// Generated by gen_resource_types
#[derive(Debug, Clone, Serialize, Deserialize, JsonHash)]
pub struct Service {
    pub name: String,
    pub namespace: String,
    // ...
}
```

#### 2. Client Function Argument Structs

For each endpoint, the generator creates argument structs for builder/convenience functions:

```rust
// Generated by gen_provider_clients
#[derive(Debug, Clone, Serialize, JsonHash)]
pub struct ListServicesArgs {
    pub namespace: String,
    pub page_size: Option<i32>,
    pub page_token: Option<String>,
}

// NOT applied to:
// - *_builder functions (they take individual params)
// - *_execute functions (they take ClientRequestBuilder)
```

**Note:** The `*_builder` and `*_execute` function naming convention means:
- Builder functions take individual parameters, not a struct
- Execute functions take `ClientRequestBuilder`
- Only convenience functions might use argument structs

### Generated Code Structure

The macro generates an `impl` block for the struct:

```rust
// Input
#[derive(JsonHash, Serialize)]
pub struct MyResource {
    pub name: String,
    pub value: i32,
}

// Output (expanded)
pub struct MyResource {
    pub name: String,
    pub value: i32,
}

impl MyResource {
    pub fn struct_hash(&self) -> String {
        use sha2::{Digest, Sha256};
        use base85rs::{encode as base85_encode};
        use serde_json::to_string;

        // Struct name
        let type_name = stringify!(MyResource);

        // Serialize to JSON
        let json = to_string(self)
            .expect("Serialize impl should not fail for valid structs");

        // Concatenate name + json
        let input = format!("{}{}", type_name, json);

        // Compute SHA-256
        let mut hasher = Sha256::new();
        hasher.update(input.as_bytes());
        let hash = hasher.finalize();

        // Encode as base85
        base85_encode(&hash)
    }
}
```

### Macro Implementation

```rust
// backends/foundation_macros/src/json_hash.rs

use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, DeriveInput, Error};

pub fn json_hash_derive(item: TokenStream) -> TokenStream {
    let ast = parse_macro_input!(item as DeriveInput);
    let struct_name = &ast.ident;

    // Check for Serialize bound - emit error if missing
    // We cannot check this at macro time, but we can document the requirement
    // The actual error will occur at compile time when serde_json::to_string fails

    let expanded = quote! {
        impl #struct_name {
            /// Compute a deterministic hash for this struct instance.
            ///
            /// The hash is computed from:
            /// 1. The name of this struct
            /// 2. The serialized JSON representation of this struct
            ///
            /// # Returns
            ///
            /// A base85-encoded SHA-256 hash string.
            ///
            /// # Panics
            ///
            /// Panics if serialization fails. This should not happen
            /// for structs that properly implement Serialize.
            pub fn struct_hash(&self) -> String {
                use sha2::{Digest, Sha256};
                use base85rs::encode as base85_encode;

                let type_name = stringify!(#struct_name);
                let json = serde_json::to_string(self)
                    .expect("JsonHash requires Serialize impl");
                let input = format!("{}{}", type_name, json);

                let mut hasher = Sha256::new();
                hasher.update(input.as_bytes());
                let hash = hasher.finalize();

                base85_encode(&hash)
            }
        }
    };

    TokenStream::from(expanded)
}
```

// backends/foundation_macros/tests/json_hash_tests.rs

use foundation_macros::JsonHash;
use serde::Serialize;

struct TestResource {

fn test_struct_hash_deterministic() {

    let b = TestResource { name: "test".into(), value: 43 };
    assert_ne!(a.struct_hash(), b.struct_hash());
}

#[test]
fn test_struct_hash_different_types_same_content() {
    #[derive(JsonHash, Serialize)]
    struct TypeA { value: i32 }
    
    #[derive(JsonHash, Serialize)]
    struct TypeB { value: i32 }
    
    let a = TypeA { value: 42 };
    let b = TypeB { value: 42 };
    // Different struct names = different hashes
    assert_ne!(a.struct_hash(), b.struct_hash());

#[test]
fn test_struct_hash_with_nested_struct() {
    #[derive(JsonHash, Serialize)]

    #[derive(JsonHash, Serialize)]

    let inner = Inner { x: 10 };
    let outer = Outer { inner: inner.clone() };

    let _ = outer.struct_hash();

## Design Decisions

### 1. Why not automatically derive Serialize?

**Decision:** User must explicitly derive `Serialize`.

**Rationale:**
- Explicit is better than implicit (Rust philosophy)
- Makes the dependency clear to readers
- Allows users to customize Serialize behavior if needed
- Compile error is clear and actionable

### 2. Why include struct name in hash?

**Decision:** Hash includes `stringify!(StructName)` + JSON.

**Rationale:**
- Distinguishes different types with identical content
- Prevents hash collisions across type boundaries
- Useful for state stores that hold multiple resource types

### 3. Why SHA-256 + base85?

**Decision:** Use existing `sha2` and `base85rs` crates.

**Rationale:**
- Already dependencies of `foundation_macros`
- Consistent with `EmbedFileAs`/`EmbedDirectoryAs` hashing
- Base85 is 20% more compact than hex encoding

### 4. Why `struct_hash()` method vs trait impl?

**Decision:** Generate inherent method, not trait implementation.

**Rationale:**
- Simpler API — no trait import needed
- No trait bounds for callers
- Self-contained generated code
- Can add more methods in the future if needed

### 5. Panic on serialization failure?

**Decision:** Use `.expect()` in generated code.

**Rationale:**
- Serialization should never fail for valid `Serialize` impls
- Keeps generated code simple
- Clear panic message if something goes wrong
- Users who need fallible hashing can wrap in `catch_unwind`

## Security Considerations

1. **Not for cryptographic purposes**: SHA-256 is cryptographically secure, but this hash is for identification, not authentication. Do not use for security-sensitive operations.

2. **No secret data**: The hash is deterministic and reversible (via JSON). Do not include secrets in hashed structs.

3. **Collision resistance**: SHA-256 provides excellent collision resistance for identification purposes.

## Performance Considerations

1. **JSON serialization**: Each hash computation requires full JSON serialization. For large structs, this may be expensive.

2. **Caching**: Users who need to hash the same value repeatedly should cache the result.

3. **State store usage**: Hash is computed at deployment time, not runtime — performance is acceptable for this use case.
