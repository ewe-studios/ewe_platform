# pi-mono Model Catalog

## File Locations

- Model types: `packages/ai/src/types.ts` (lines 394-417)
- Model catalog: `packages/ai/src/models.generated.ts` (~14,830 lines)
- Model utilities: `packages/ai/src/models.ts`
- Model resolution: `packages/coding-agent/src/core/model-resolver.ts`

## Model Interface

```typescript
interface Model<TApi> {
  id: string;           // e.g., "claude-opus-4-5"
  name: string;         // e.g., "Claude Opus 4.5"
  api: TApi;            // which API type to use
  provider: Provider;   // which provider namespace
  baseUrl: string;      // API endpoint URL
  reasoning: boolean;   // supports thinking/reasoning
  input: ("text" | "image")[];  // supported input modalities
  cost: {               // $ per million tokens
    input: number;
    output: number;
    cacheRead: number;
    cacheWrite: number;
  };
  contextWindow: number;    // max context tokens
  maxTokens: number;        // max output tokens
  compat?: OpenAICompletionsCompat | OpenAIResponsesCompat;  // compat overrides
}
```

## Auto-Generated Catalog

The model catalog (`models.generated.ts`) is auto-generated by `scripts/generate-models.ts`. This script:
1. Fetches model lists from provider APIs (OpenRouter, Anthropic, OpenAI, etc.)
2. Merges with manually curated model entries
3. Applies pricing data from official sources
4. Generates compatibility flags for OpenAI-compatible endpoints
5. Writes the consolidated TypeScript file

The catalog contains hundreds of models across all providers. Each model entry includes pricing, context window, max tokens, and API routing information.

## Cost Calculation

```typescript
// packages/ai/src/models.ts lines 39-46
usage.cost.input = (model.cost.input / 1000000) * usage.input;
usage.cost.output = (model.cost.output / 1000000) * usage.output;
usage.cost.cacheRead = (model.cost.cacheRead / 1000000) * usage.cacheRead;
usage.cost.cacheWrite = (model.cost.cacheWrite / 1000000) * usage.cacheWrite;
usage.cost.total = usage.cost.input + usage.cost.output + usage.cost.cacheRead + usage.cost.cacheWrite;
```

Pricing is per million tokens. The model descriptor carries the pricing data, so cost calculation is a simple multiplication at the point of usage tracking.

## Context Token Estimation

```typescript
// packages/coding-agent/src/core/compaction/compaction.ts
estimateContextTokens(messages): number
```

The estimation strategy:
1. Uses the last assistant message's `usage.totalTokens` as an anchor
2. Estimates tokens for any messages after that point using a character-based estimator
3. Returns the total estimated context token count

This is used to determine when to trigger compaction:

```
trigger when: estimated_tokens > contextWindow - reserveTokens - keepRecentTokens
```

## Model Resolution

The model resolver (`packages/coding-agent/src/core/model-resolver.ts`) determines which model to use based on:
1. CLI arguments (`--provider`, `--model`)
2. Settings (project-local or global)
3. Existing session (when continuing/resuming)
4. Default fallback

The resolution order:
1. If `--model` is specified, look up in the catalog
2. If a session is being continued, use the session's model
3. If settings specify a default, use that
4. Fall back to a sensible default (typically Claude Sonnet)

## Scoped Models for Cycling

The `--models` flag accepts a comma-separated list of model IDs. When multiple models are specified:
- The agent tries the first model
- On certain errors (rate limits, context overflow, etc.), it cycles to the next model
- This provides automatic fallback without manual intervention

## Compatibility Overrides

The `compat` field on models allows per-model compatibility overrides for OpenAI-compatible endpoints:

```typescript
interface OpenAICompletionsCompat {
  supportsStrictMode?: boolean;   // Whether strict schema mode is supported
  requiresToolResultName?: boolean; // Whether tool results need a name field
  // ... other compat flags
}
```

This is critical because different OpenAI-compatible providers have subtle differences:
- Some support `strict: true`, some don't
- Some require `name` on tool result messages
- Some return `reasoning_content` in different fields
- Some use different cache control formats

## Key Patterns for foundation_ai

1. **Auto-generated model catalog** keeps pace with provider API changes without manual updates
2. **Per-model pricing** in the descriptor enables instant cost calculation
3. **Compatibility overrides** handle provider-specific quirks without branching the provider code
4. **Model cycling** with `--models` provides automatic fallback
5. **Context token estimation** using last known usage + character-based estimation is pragmatic
6. **Model-as-descriptor** pattern means all routing decisions are data-driven
